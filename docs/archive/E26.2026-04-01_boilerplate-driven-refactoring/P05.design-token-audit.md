# P05: 디자인 토큰 점검

## 목표
- 토큰 계층이 DT-01-02(Primitive → Semantic → Component) 3-tier 구조를 따르며, 각 계층의 역할이 명확하다
- 토큰 네이밍이 DT-02(Namespace → Object → Base → Modifier) 규칙을 준수한다
- UI 코드에서 하드코딩된 원시값(HEX/RGB/px 등)이 DT-04-01/02 기준으로 0건이다
- 토큰 추가/변경 거버넌스(DT-04-03/04)가 문서화되어 있다

## 구현 상세

### 접근

1. **3-Tier 계층 점검 (DT-01-02, DT-01-03, DT-01-04)**
   - `src/domain/constants/`, `src/domain/tokens/` 구조가 Primitive/Semantic/Component 계층으로 분리되어 있는지
   - 토큰 소비 흐름이 단방향(Primitive → Semantic → Component → UI)인지 확인
   - `src/domain/tokens/global/`, `typography/`, `ui/` 디렉토리가 어느 계층에 해당하는지 매핑

2. **네이밍 규칙 점검 (DT-02-01~05)**
   - CSS 변수명에서 `--ds-` / `--ui-` 접두사 사용 여부 전수 스캔
   - 접두사 없는 CSS 변수(`--` 로 시작하되 `--ds-`/`--ui-` 아닌 것) 식별
   - kebab-case 위반 여부 (camelCase, snake_case 혼용)
   - Namespace → Object → Base → Modifier 순서 준수 여부 샘플링

3. **타입별 템플릿 점검 (DT-03-01~04)**
   - Color 토큰: `--ds-color-{scale}-{step}` (Primitive), `--ds-color-{role}-{name}` (Semantic) 패턴 준수
   - Spacing/Radius/Border: `--ds-spacing-{key}`, `--ds-radius-{key}` 패턴 준수
   - Typography: `--ds-text-{style}-{property}`, `--ds-font-{name}` 패턴 준수
   - 기타: duration, ease, shadow, size, z-layer 접두사 사용 여부

4. **하드코딩 원시값 스캔 (DT-04-01, DT-04-02)**
   - `src/app/` 내 `.tsx`, `.css`, `.module.css` 파일에서 HEX(`#xxx`), RGB(`rgb(`), 고정 px 값 검색
   - `src/domain/constants/`, `src/domain/tokens/` (토큰 정의 영역)은 원시값 허용 — 제외
   - `src/shared/styles/` 글로벌 CSS에서의 원시값 사용 패턴 확인

5. **거버넌스 점검 (DT-04-03, DT-04-04)**
   - 토큰 추가/변경/폐기 절차가 문서화되어 있는지 (yamang-design-stack 또는 docs/design/ 내)
   - 호환성 레이어(compatibility layer) 전략이 있는지

### 변경 범위
- **수정 없음** — 점검 결과를 본 문서에 기록
- 위반 사항 발견 시 수정 방향을 권장 사항으로 기록

## 점검 결과 (2026-04-01)

### 1. 계층 구조 (DT-01)

- **`src/domain/tokens/`:** `global/`(spacing, colors, motion 등), `typography/`, `ui/`로 나뉘어 있고 `index.ts`에서 re-export — **Primitive·세부 축 분리**에 해당.
- **`docs/design/03-token-architecture.md`:** External Input → Resolved Palette → Primitive → Semantic → Theme → Component 흐름을 **문서상 5~6단**으로 서술. 엄격한 “폴더명 = 3-tier”와 1:1이 아니라 **개념 계층은 문서·코드 모두 존재**로 판단.
- **`domain/constants/`**, **`domain/styles/presets/`:** 시맨틱·스타일 프리셋과 상수 — Semantic/Theme 층과 연계.
- **소비:** Lab/빌드 페이지 CSS는 대량 `var(--ds-*)` 사용. 아트 갤러리 페이지는 **의도적 팔레트**로 원시 색 혼용(아래 4).

### 2. 네이밍·접두사 (DT-02, DT-03)

- 디자인 시스템 구역: `--ds-*` 위주, `.cursor/skills/yamang-design-stack/SKILL.md`의 `--ds-{category}-...` 패턴과 **정합**.
- **`--ui-*`:** Shell/고정 UI 규칙은 스킬에 명시; 실제 사용은 `shared/styles`·일부 페이지에서 확인 가능.
- **kebab-case:** CSS 변수 스캔 샘플에서 camelCase 변수명 다수 발견 없음.

### 3. 하드코딩 원시값 (DT-04-01/02)

- **`src/app/pages/art/`** (MonetWaterLilies, `_shared`, Matisse, Mondri 등): **HEX·rgba 다수** — 아트 페이지 전용 브랜드 색·글래스 효과. `yamang-design-stack`의 “하드코딩 금지”와 **충돌**하나, **예외로 기록** (디자인 복제 목적, P03 Sonar 대비 Accepted와 동일 정책 축).
- **Lab·components:** 상대적으로 `var(--ds-*)` 비중 높음; 완전 0건 원시값은 아님(일부 px·fallback).

### 4. 거버넌스 (DT-04-03/04)

- **`.cursor/skills/yamang-design-stack/SKILL.md`:** Styling Policy, Token Naming, 하드코딩 예외 조건, `npm run check:literals` — **프로젝트 단위 거버넌스**로 기능.
- **`docs/design/03-token-architecture.md`**, **`18-token-naming-reference.md`** 등: 계층·네이밍 참조.
- **신규 토큰 전용 체크리스트 단일 파일**은 없음 — 스킬 + design 문서 조합으로 **부분 충족**으로 기록.

### 권장 후속 (코드 변경 없음, 백로그 후보)

1. 아트 페이지 원시색을 토큰/프리셋으로 점진 이전할지 정책 결정.
2. `check:literals` CI 연동 여부 검토.
