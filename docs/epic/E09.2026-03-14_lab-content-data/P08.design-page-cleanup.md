# P08: 디자인 페이지 정비

## 목표

디자인 설정 페이지(DesignSettingsLab)의 프리셋 탭·시맨틱 매핑·공통 UI를 정비하여 데이터 표시 일관성, 상태 동기화, UX를 개선한다.

---

## 1) 프리셋 탭 — 스타일 미지정 데이터가 글라스모피즘으로 보이는 이유 (조사 결과)

### 코드 경로

- **마이그레이션**: `src/components/GlobalSettings/types.ts`의 `migrateV1ToV2()`는 v1의 `styleName`을 그대로 복사한다. v1에 `styleName`이 없으면 `undefined`가 v2에 들어간다.
- **기본값 사용처**:
  - `App.tsx`: `initialStyleName={initialSettings?.styleName ?? 'minimal'}` → 저장값이 없으면 `'minimal'`.
  - `ThemeProvider.tsx`: `initialStyleName = initialTheme ?? 'minimal'`.
  - `PresetTab.tsx` Custom 카드: `preset.settings.styleName ?? 'minimal'` → 표시만 기본값, 저장값은 그대로.

따라서 **코드상 기본 스타일은 모두 `'minimal'`**이다. 글라스모피즘이 나오는 경우는 다음 정도만 가능하다.

- **과거에 실제로 `styleName: 'glassmorphism'`으로 저장된 데이터**가 있는 경우.
- **빌트인 프리셋 카드의 스타일 배지**: Default/Natural/Pop 카드는 **각 프리셋에 스타일이 없고**, 현재 draft의 `styleName`을 배지에 표시한다(`PresetTab.tsx` 398행 근처). 따라서 현재 선택 스타일이 글라스모피즘이면 **모든 빌트인 카드 배지가 “Glass”로 보일 수 있다.**

### 권장 조치

- v2 마이그레이션 시 `styleName`이 `undefined`이거나 유효한 `StyleName`이 아니면 `'minimal'`로 보정해 저장/표시하도록 `migrateV1ToV2` 또는 로드 직후 정규화를 추가할지 검토.
- 빌트인 프리셋은 “팔레트만 있는 데이터”이므로, 카드 배지에 “현재 draft 스타일” 대신 “스타일 없음” 또는 “공통” 같은 라벨을 쓰거나, 배지를 숨기는 방안 검토.

## 2) 프리셋 탭 — “모든 프리셋” 서브탭

### 요구사항

- 프리셋 탭 내에 **모든 프리셋을 한 목록으로 보여주는 서브탭**을 둔다.
- 현재: Custom / Default / Natural / Pop 등 **카테고리별 탭**만 있음 (`ThemeTabNavigation` + `themeRegistry`).

### 구현 방향

- 서브탭에 **“전체” 또는 “모든 프리셋”** 항목 추가.
- `palettes/presets/registry.ts`의 `getAllThemes()`를 사용해 카테고리 구분 없이 플랫 리스트로 렌더링.
- 검색(`ThemeSearchBar`)은 “전체” 탭에서도 동일하게 적용되도록 유지.

## 3) 프리셋 탭 — 컬러셋 출처, 활성 표시, 좌측 팔레트 미갱신, 상태 관리

### 컬러셋 출처

- **좌측 컬럼 “컬러 팔레트”**: `useGlobalSettings`의 `palette` = `localPalette` = `extractPaletteFromSelection(localSelection)`.
- **`extractPaletteFromSelection`** (`useGlobalSettings.ts` 98–100행):
  - `selection.type === 'custom'` → `selection.colors` 반환.
  - **`selection.type === 'preset'`** → **`DEFAULT_PALETTE` 반환** (presetId로 해석하지 않음).
- 따라서 **빌트인 프리셋을 선택해도 좌측 컬럼은 항상 기본 팔레트**만 보인다. 프리셋 전용 컬러셋은 `palettes/presets/registry` + `findThemeById(presetId)` 등으로 해석하는 경로가 **좌측 표시에는 사용되지 않음**.

### 활성 프리셋 표시 방식

- 현재: “현재 적용됨”일 때 **체크 아이콘** (`appliedBadge` + `<Icon name="check" />`).
- 요구: **체크 대신 배경색 변경 또는 border**로 활성 표시.

### 좌측 컬러 팔레트가 적용된 프리셋에 맞춰 갱신되지 않는 이유

- **원인**: `extractPaletteFromSelection(selection)`이 `type === 'preset'`일 때 presetId를 사용하지 않고 `DEFAULT_PALETTE`만 반환함.
- **해결 방향**: `useGlobalSettings`에서 “표시/편집용 팔레트”를 계산할 때, `selection.type === 'preset'`이면 **presetId로 정의를 조회한 뒤 해당 definition의 colors를 사용**하도록 변경. (예: `resolveSelection(selection, …)` 또는 `findThemeById(selection.presetId)` 결과의 `colors`를 사용.)

### 상태가 중앙/최상위에서 관리되는지

- **설정 draft**: `useGlobalSettings`(DesignSettingsLab 내부)에서 **로컬 상태**로 관리하고, “적용” 시에만 `ThemeProvider`/App의 `appliedSettings`와 localStorage에 반영.
- **테마/팔레트 해석**: `ThemeProvider` + `usePaletteSelection`이 **최상위**에서 selection → definition/colors 해석을 담당.
- **불일치**: DesignSettingsLab의 “좌측 팔레트”는 **해석된 팔레트가 아니라** `extractPaletteFromSelection`만 쓰므로, preset 선택 시 **중앙 해석 결과와 동기화되지 않음**. 위 “해결 방향”으로 **해석 결과를 한 곳에서 쓰도록** 맞추면, 중앙/최상위 관리와 일치하게 됨.

## 4) 프리셋 이름 입력·저장 위치

### 요구사항

- **프리셋 이름 입력 + 저장**을 **좌측 컬럼의 “컬러 팔레트” 섹션 위**에 둔다.

### 현재

- `PresetTab.tsx`: 저장 입력창과 “저장” 버튼이 **우측 프리셋 브라우저 하단** `saveInline`에 있음.

### 구현

- 좌측 컬럼 상단에 “프리셋 이름” 입력 + “프리셋으로 저장” 버튼 블록을 추가하고, 기존 `saveInline`은 제거하거나 우측에는 “저장” 버튼만 유지(선택). 좌측 배치는 “컬러 팔레트” `<section>` 바로 위.

## 5) 시맨틱 매핑 내부 가져오기/내보내기 제거 — 완전한 하나의 세트로 통일

### 요구사항

- 시맨틱 매핑 탭 **내부**의 가져오기/내보내기는 제거한다.
- 가져오기/내보내기는 **항상 완전한 하나의 세트**(전역 설정 + 시맨틱 매핑 + 컴포넌트 매핑 등)만 사용한다.

### 현재

- **페이지 공통 footer**: 내보내기 / 가져오기 / 초기화 / 적용 → `createGlobalSettingsPayload` 등으로 **전체 설정** export.
- **시맨틱 탭 상단**: “가져오기” / “내보내기” 버튼이 있음 (`SemanticTab.tsx`).  
  - 내보내기: `createSemanticMappingPayload` + `downloadYamangJSON` → **시맨틱 매핑만** 저장하는 파일 생성.  
  - 가져오기: 파일 선택 후 `extractSemanticOverrides(parsed)` 또는 `parseYamangJSON`으로 **시맨틱 오버라이드만** 적용.

### 구현

- `SemanticTab`에서 **가져오기/내보내기 버튼 및 관련 핸들러·hidden input** 제거.
- “오버라이드 초기화”만 유지.
- 전체 가져오기/내보내기는 **페이지 footer**만 사용하고, 기존 `importSettings`/`exportSettings`가 “완전한 세트”를 처리하는지 확인 후 유지.

## 6) 디자인 설정 페이지 양식(배지·카드) 및 공통 푸터 아이콘 버튼화

### 양식(배지, 카드) 구성 확인

- **배지**: `PresetTab.module.css` — `.presetBadge`, `.presetBadge.yamang`, `.presetBadge.natural`, `.presetBadge.custom`, `.styleBadge`, `.appliedBadge`.
- **카드**: `.presetItem`, `.flatCard`, `.paletteRow`, `.settingRow` — `var(--shell-bg-subtle)`, `var(--shell-border-default)` 등 셸 토큰 사용.
- **디자인 설정 페이지 공통**: `DesignSettingsLab.module.css` — `.footer`, `.footerBtn`, `.applyBtn` 등. 배지/카드는 PresetTab·SemanticTab·ComponentMappingTab 각 모듈 CSS에서 정의.

### 공통 푸터 아이콘 버튼화

- **요구**: 공통 푸터를 **아이콘 버튼**으로 줄여서 **“디자인 시스템 설정” 타이틀과 같은 행의 우측**에 배치.
- **현재**: 타이틀은 `.headerRow` 안 `.title`, 푸터는 `.footer`에 별도 행으로 “내보내기 / 가져오기 / 초기화” + “적용” 버튼이 텍스트+아이콘으로 있음.
- **구현**:  
  - `.headerRow`에 타이틀(h1)과 우측 액션을 한 줄로 배치.  
  - “내보내기” “가져오기” “초기화”는 **아이콘만** 보이게 하고 `aria-label`로 접근성 유지.  
  - “적용” 버튼은 아이콘만 또는 아이콘+짧은 텍스트로 동일 행 우측에 배치.  
  - 기존 `.footer`는 제거하거나, 필요 시 탭 콘텐츠 하단의 보조 액션만 남김.

## 7) 빌트인 프리셋 데이터 구조 정비 (현황 + 기본값 검토)

이번 정비에서 **기존 빌트인 프리셋의 데이터 구조를 완전히 정비**하려고 한다. 먼저 현재 프리셋 내용과 데이터 흐름을 정리하고, **기본값을 어디서 어떻게 쓸지**를 정한 뒤 구현에 들어간다.

### 7.1 현재 빌트인 프리셋 목록 및 내용

| 위치 | id (metadata.id) | name | colors | bgStrategy | semanticMapping |
|------|------------------|------|--------|------------|-----------------|
| `palettes/presets/default/default.ts` | `default` | default | primary, secondary, accent, sub, neutral 모두 명시 | light | 없음 (bgStrategy 기본 매핑 사용) |
| `palettes/presets/natural/SpringCreamSoda01.ts` | `springCreamSoda01` | springCreamSoda01 | 5색 모두 명시 | light | 없음 |
| `palettes/presets/pop/OrientalChineseRestaurant01.ts` | `orientalChineseRestaurant01` | orientalChineseRestaurant01 | 5색 모두 명시 | light | 있음 (bg, text, border 오버라이드) |

- **타입**: 모두 `PaletteDefinition` (`palettes/types.ts`).
- **공통 필드**: `name`, `subname`, `colors`, `bgStrategy`, `metadata` (id, displayName, category, description). `contrast`는 default·SpringCreamSoda01에만 있음.
- **스타일/시스템**: 빌트인 프리셋에는 **styleName·systemPreset이 없음**. “팔레트 정의”만 있고, 스타일은 적용 시점의 전역 설정(StoredSettings)에서 가져옴.

### 7.2 데이터 흐름 (현재)

1. **정의 소스**: `src/palettes/presets/{default|natural|pop}/*.ts` → 각 파일이 `PaletteDefinition` export.
2. **레지스트리**: `palettes/presets/registry.ts`가 카테고리별로 `themeRegistry` (ThemeGroup[]) 구성. `getAllThemes()`로 플랫 리스트.
3. **ID 매핑**: `constants/theme-presets.ts`에서 `getAllThemes()` → `themePresets = Record<metadata.id, PaletteDefinition>`. `metadata.id` 없으면 에러.
4. **해석**: `themes/presets.ts`의 `palettePresets` = `themePresets`. `usePaletteResolution` / `resolveSelection`에서 `selection.type === 'preset'`일 때 `palettePresets[presetId] ?? palettePresets.default`로 definition 조회 후 `definition.colors`를 ExternalPalette로 사용.

즉, **빌트인 프리셋 = PaletteDefinition 한 개**가 곧 “팔레트 선택지” 하나이고, 스타일/시스템은 별도 레이어다.

### 7.3 기본값이 쓰이는 지점 (고민용)

| 계층 | 기본값 사용처 | 현재 동작 |
|------|----------------|-----------|
| **팔레트 색상** | `utils/palette.ts` `resolvePalette()` | `secondary`/`accent`/`neutral`/`sub` 없으면 primary 기반 파생 (deriveSecondary, deriveAccent, deriveNeutral, deriveSub). |
| **시맨틱 매핑** | `palettes/index.ts` `createPalette()` | `defaultSemanticMappings[definition.bgStrategy]`를 베이스로 두고, `definition.semanticMapping`을 `getMergedMapping`으로 병합. 프리셋에 semanticMapping 없으면 전부 bgStrategy 기본값. |
| **설정(draft)** | `useGlobalSettings`, `App`, `ThemeProvider` | styleName/systemPreset/semanticMapping 등은 각자 기본값(minimal, default, null) 보정. |
| **빌트인 프리셋 표시** | PresetTab 카드 | 스타일 배지는 “현재 draft의 styleName”을 표시 (프리셋 자체에는 스타일 없음). |

**고민할 점:**

- **팔레트 스키마**: 지금은 `colors`에 optional이 많고, 런타임에 `resolvePalette`가 채움. “완전한 구조”로 갈 때 “빌트인은 5색 전부 명시”로 통일할지, “optional 유지 + 파생 규칙 문서화”로 갈지 결정 필요.
- **시맨틱**: “없음 = bgStrategy 기본 매핑”이 명확하므로, 빌트인은 `semanticMapping` 생략 가능 유지해도 됨. 다만 “일부만 오버라이드” 시 스키마를 어떻게 제한할지(전부 쓰기 vs Partial)는 정리해 두면 좋음.
- **스타일/시스템**: 빌트인은 팔레트만 갖고, style/system은 “현재 사용자 설정”을 쓰는 현재 방식 유지할지, 아니면 “빌트인별 기본 styleName”을 옵션으로 넣을지(예: default = minimal, pop = brutalism) 정책 결정 필요.
- **기본값 일원화**: “기본값”을 쓰는 곳이 resolvePalette / defaultSemanticMappings / StoredSettings 초기값 등 여러 군데라, “기본값 정의”를 한 문서 또는 한 모듈로 모을지 검토.

### 7.4 구조 변경 방향 (구현 전 설계)

- **단일 소스**: 빌트인도 JSON 등 데이터 파일로 옮겨서 E09 콘텐츠 디렉터리와 형식 통일할지, 아니면 TS export 유지할지.
- **스키마 통일**: `PaletteDefinition` (및 ExternalPalette)에서 필수/선택/기본값 규칙을 문서로 고정하고, 빌트인 프리셋이 그 규칙을 만족하도록 수정.
- **ID·이름 규칙**: `metadata.id`는 그대로 “프리셋 선택 키”. `name`(내부) / `displayName`(UI) 역할 명확히.
- **사용자 프리셋과의 관계**: 사용자 저장 프리셋(StoredPreset)은 “전체 설정 스냅샷”(palette + style + system + semanticMapping). 빌트인은 “팔레트 정의만”. 이 이원 구조 유지할지, 빌트인도 “기본 스타일”을 가진 스냅샷으로 확장할지 결정 후, 그에 맞춰 타입/기본값 정리.

### 7.5 추천 방향

| 항목 | 추천 | 이유 |
|------|------|------|
| **팔레트 스키마** | **빌트인은 5색 전부 명시 유지**, 런타임은 optional + `resolvePalette` 유지 | 빌트인은 수가 적고 의도가 명확하므로 전부 명시하면 검수·비교가 쉽다. 사용자 입력·가져오기에는 optional + 파생이 유리하므로 런타임 규칙은 그대로 둔다. |
| **시맨틱** | **생략 = bgStrategy 기본 매핑 유지**. Partial 오버라이드만 문서화 | OrientalChineseRestaurant01처럼 “일부만 오버라이드”가 이미 쓰이고, `getMergedMapping`으로 잘 동작하므로 스키마는 유지. README나 타입 주석에 “경로별 ScaleReference/직접색 규칙”만 정리. |
| **스타일/시스템** | **빌트인은 팔레트만**, 스타일은 “현재 사용자 설정”만 사용 | 빌트인에 styleName을 넣으면 “선택 시 스타일까지 덮어쓴다”는 동작이 추가되어 혼란과 마이그레이션 부담이 생김. “팔레트 선택 = 색만, 스타일은 전역”이 단순하고 예측 가능하다. 카드 배지는 “스타일 없음” 또는 배지 숨김으로 처리 (1) 항과 일치). |
| **기본값 일원화** | **한 문서로 정리**, 코드는 기존 위치 유지 | `docs/` 또는 P08 하위에 “기본값 정책” 한 페이지: resolvePalette 파생 규칙, bgStrategy별 기본 매핑 요약, StoredSettings 초기값 표로 정리. 코드를 한 모듈로 모을 필요는 없고, “기본값은 여기 문서 참고”만 명시. |
| **단일 소스(형식)** | **당분간 TS 유지**, JSON은 “추후 옵션” | 빌트인 개수가 많지 않고, 타입·import가 바로 되며, 이미 카테고리별 index로 정리되어 있다. E09 콘텐츠는 랩 설명용 JSON이고, 프리셋은 “코드에 가까운 설정”이므로 TS가 적합. 나중에 에디터/비개발자 편집이 필요해지면 그때 JSON + 로더 도입 검토. |
| **빌트인 vs 사용자** | **이원 구조 유지**: 빌트인 = 팔레트 정의만, 사용자 = 전체 스냅샷 | 레이어가 명확하고(팔레트 / 스타일 / 시맨틱 / 컴포넌트), “빌트인 선택 → 적용”이 “색만 바꾸고 스타일·시맨틱은 기존 설정 유지”로 동작하는 현재 모델이 직관적이다. |

**정리**: 기본값·스키마·표시 정책만 문서로 고정하고, 빌트인 프리셋은 **현재 TS 구조를 유지한 채** 필수 필드(metadata.id, colors 5색)와 선택 필드(semanticMapping Partial) 규칙만 명시하면 된다. “완전한 구조로 바꾼다”는 것을 “JSON 이전”이 아니라 “스키마·기본값 규칙을 문서화하고 코드가 그 규칙을 따르게 정리”로 해석하는 것을 추천한다.

위 7.3·7.4를 팀에서 정한 뒤, “빌트인 프리셋 데이터 구조 완전 변경” 구현을 진행한다. **7.5 추천을 따를 경우** 구현량은 “문서 추가 + (선택) 타입 주석/검증” 수준으로 줄어든다. 구현 체크리스트는 7) 설계 확정 후 세부 항목 추가.

## 8) 설정 통합 관리 — 한 페이지·한 체계

모든 설정이 디자인 설정 페이지 하나로 모였지만, **저장소·상태·진입로가 나뉘어 있어** 관리가 복잡하다. 아래처럼 **“현재 디자인 시스템 상태”를 하나로 묶고, 그걸 유일한 소스로 쓰면** 단순해진다.

### 8.1 현재 복잡도 (원인)

| 구분 | 저장 키 / 상태 | 담당 | 비고 |
|------|----------------|------|------|
| 현재 적용 설정 | `yamang-design-settings` | App, useGlobalSettings | StoredSettings (palette, style, system, semanticMapping) |
| 사용자 저장 프리셋 | `yamang-design-presets` | useGlobalSettings | StoredPreset[] |
| 팔레트 선택 | `yamang-palette-selection` | ThemeProvider, palette-selection.ts | PaletteSelection (presetId 또는 custom colors) |
| 컴포넌트 매핑 오버라이드 | `yamang-component-mapping-overrides` | component-mapping-storage.ts | Design Settings 적용 시 함께 초기화 가능 |
| 커스텀 시맨틱 프리셋 | `yamang-custom-semantic-presets` | useCustomSemanticPresets, PaletteLab | 시맨틱 “저장 프리셋” (별도 개념) |

- **중복**: 적용 시 `saveStoredSettings(draft)`와 `setPaletteSelection(...)` → `savePaletteSelection(...)` 둘 다 호출되어, **팔레트 정보가 두 키에 동시에** 들어감.
- **초기값 불일치 가능성**: App은 `yamang-design-settings`에서 로드해 ThemeProvider에 `initialPalette` 등을 넘기지만, ThemeProvider는 `initialSelection`이 없으면 `loadPaletteSelection()`으로 **다른 키**를 참조할 수 있음 (현재는 App이 항상 palette를 넘겨서 실사용에서는 한 소스지만, 코드 경로가 둘).
- **진입로 분리**: “현재 팔레트”를 바꾸는 곳이 Design Settings(적용)와 PaletteLab(선택 후 저장) 등으로 나뉨.

### 8.2 통합 방향: “현재 상태 = 한 덩어리”

- **단일 “현재 디자인 시스템 상태”**
  - 타입 하나로 통일: 예) `DesignSystemState` = 지금의 `StoredSettings` + `componentMapping` (선택).  
  - 필드: `palette`, `palettePresetId`, `styleName`, `systemPreset`, `semanticMapping`, `componentMapping?(선택)`, `updatedAt` 등.
- **저장소 하나**
  - “현재 상태”는 **한 키**에만 저장: `yamang-design-settings` (이름 유지해도 됨).  
  - `yamang-palette-selection`은 **제거**하거나, 마이그레이션용 읽기만 하고 쓰기는 중단.  
  - 적용/초기화/가져오기 시 **이 키만** 읽고 쓴다.
- **PaletteSelection은 “파생값”**
  - `DesignSystemState`에서만 복원: `palettePresetId`가 있으면 `{ type: 'preset', presetId }`, 없으면 `{ type: 'custom', colors: palette }`.  
  - ThemeProvider는 “현재 상태”를 App에서 받아, 그걸로 selection을 만들고 내부 state를 초기화. 별도 `loadPaletteSelection()`은 쓰지 않음.
- **사용자 프리셋**
  - `yamang-design-presets`는 “현재 상태 스냅샷” 배열로 유지. “불러오기” = 해당 스냅샷으로 현재 상태를 덮고, 위 단일 키에 저장 + 적용.
- **가져오기/내보내기**
  - 항상 “현재 상태 전체”(+ 필요 시 사용자 프리셋 목록) 한 세트만 다룸. (5)번과 동일.)

그러면 **디자인 설정 페이지**는 “그 한 덩어리를 편집하는 UI”이고, 적용은 “그 덩어리를 저장 + ThemeProvider에 반영” 한 경로만 있으면 된다.

### 8.3 커스텀 시맨틱 프리셋 처리

- `yamang-custom-semantic-presets`는 “팔레트 + 시맨틱 오버라이드 저장”용이라, “현재 상태”와는 성격이 다름.
- **옵션 A**: 그대로 별도 키·별도 UI(PaletteLab) 유지. Design Settings에서는 “현재 상태”만 다루고, 시맨틱 탭은 그 상태의 `semanticMapping`만 편집.
- **옵션 B**: 나중에 “사용자 프리셋”과 통합해 “이름 있는 스냅샷”이 “팔레트+시맨틱” 또는 “전체 상태” 중 어떤 걸 저장하는지 타입으로 구분. 당장은 A로 두고, 통합은 추후 검토해도 됨.

### 8.4 구현 시 유의점

- **마이그레이션**: 기존 `yamang-palette-selection` 사용처를 제거하기 전에, 앱 최초 로드 시 “design-settings 없고 palette-selection만 있으면 → 그걸로 StoredSettings 형태를 만들어 design-settings에 한 번 저장”하는 로직을 넣으면, 기존 사용자 환경을 한 번에 단일 소스로 이전할 수 있음.
- **ThemeProvider**: `initialSelection`을 App이 “현재 상태”에서 파생해 넘기도록 하고, `loadPaletteSelection()` fallback은 “design-settings 없을 때만” 쓰거나 제거.

### 8.5 팔레트 프리셋 ID 정책 — 유지할까?

**현재 역할**: `StoredSettings.palettePresetId`는 “빌트인 프리셋을 선택했을 때” 그 프리셋의 id를 저장한다. 저장 시 `palette`도 함께 들어가지만, preset 선택 시 `extractPaletteFromSelection`이 preset을 해석하지 않아 **palette 필드가 잘못 채워지는 버그**가 있음(3)번). 즉, “참조(presetId) + 스냅샷(palette)” 이원 구조인데, 스냅샷이 preset인 경우 제대로 안 채워지는 상태.

**옵션 A — ID 유지 (참조 + 스냅샷)**  
- 저장: `palettePresetId`(있으면) + `palette`(항상, preset이면 해석해서 채움).  
- 복원: 로드 시 presetId 있으면 selection = `{ type: 'preset', presetId }`, 없으면 `{ type: 'custom', colors: palette }`.  
- 장점: “어떤 빌트인에서 왔는지” 보존 가능. 나중에 빌트인 정의를 바꾸면 “현재 상태를 해당 프리셋 최신판으로 다시 맞추기” 같은 UX 가능.  
- 단점: 저장/로드/export 시 “참조 vs 스냅샷” 두 경로를 항상 맞춰야 하고, 해석 로직이 여러 곳에 흩어짐.

**옵션 B — ID 제거, 항상 스냅샷만**  
- 저장: **팔레트는 항상 `palette`(ExternalPalette)만** 저장. `palettePresetId`는 저장하지 않음.  
- 복원: “현재 팔레트”는 항상 저장된 `palette`로만 복원. selection은 “저장된 palette와 동일한 내용인 빌트인이 있으면 그 id로 표시”하는 식으로 **UI용으로만** 계산 가능(선택).  
- 장점: 한 가지 표현만 다루면 되어 단순함. export/import도 “항상 풀 스냅샷”이라 해석 불필요. 8) 통합 방향(한 덩어리)과 잘 맞음.  
- 단점: “이 상태가 Default 프리셋에서 왔다”는 정보는 저장하지 않음. 빌트인 정의를 나중에 수정해도, 이미 저장된 상태는 자동으로 따라가진 않음(스냅샷이므로).

**추천 — 옵션 B (ID는 저장하지 않음)**  
- “설정 통합”과 “한 페이지·한 체계”를 위해서는 **저장되는 현재 상태는 “항상 풀 스냅샷”(palette + style + semantic + …)** 이 하나로 있는 편이 관리하기 쉽다.  
- `palettePresetId`를 **저장하지 않고**, 빌트인 선택 시에는 “그 프리셋의 colors를 복사해 `palette`에 넣고 저장”하면 된다.  
- “지금 색이 Default와 같다” 같은 표시가 필요하면, **UI 전용**으로 “저장된 palette와 동일한 빌트인이 있는지” 비교해 배지만 붙이면 됨.  
- 정리: **팔레트 프리셋 ID는 “저장 정책”에서는 제거하고, “현재 팔레트 = 항상 palette 스냅샷”만 유지하는 것을 추천.** 기존에 `palettePresetId`만 있고 palette가 비었거나 잘못된 데이터는 마이그레이션 시 presetId로 한 번 해석해 palette를 채운 뒤, 이후부터는 palette만 저장하면 됨.

이렇게 하면 **한 페이지 = 한 “현재 상태” = 한 저장소 = 한 적용 경로**로 정리되어, 관리가 훨씬 단순해진다.
