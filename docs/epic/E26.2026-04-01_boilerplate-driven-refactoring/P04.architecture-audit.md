# P04: 아키텍처 점검

## 목표
- YamangDesign의 import 그래프가 AR-01-02(의존 방향) 규칙을 준수하며, 역방향 import가 0건이다
- 현재 3-layer 구조(domain/app/shared)의 선정 근거가 AR-01-01 기준으로 문서화되어 있다
- 컴포넌트 분류 체계가 AR-02-01 기준으로 명시되어 있고, 실제 배치와 일치한다
- 상태 관리(Context/hook)의 레이어 배치가 AR-02-02/AR-02-03에 부합한다

## 구현 상세

### 접근

1. **의존 방향 전수 스캔 (AR-01-02, AR-02-02)**
   - `src/domain/` 내 모든 파일에서 `@app/` import 여부를 grep — 역방향 위반
   - `src/shared/` 내 모든 파일에서 `@domain/` 또는 `@app/` import 여부를 grep — 역방향 위반
   - 허용 방향: `app → domain → shared` (단방향만)

2. **모듈 경계 점검 (AR-01-02)**
   - 레이어별 barrel export(`index.ts`) 존재 여부 확인
   - barrel을 우회하여 내부 파일을 직접 import하는 패턴 식별

3. **아키텍처 선정 근거 확인 (AR-01-01, AR-02-01)**
   - `docs/design/ARCHITECTURE.md` 또는 동등 문서에 3-layer 선정 이유가 기록되어 있는지 확인
   - AR-02-01이 요구하는 두 축(코드 구조, UI 컴포넌트) 선택이 명시되어 있는지 확인

4. **컴포넌트 분류 체계 점검 (AR-02-01)**
   - `src/app/components/`의 분류 기준(Atoms/Molecules/Organisms 또는 동등)이 문서화되어 있는지
   - 실제 컴포넌트 배치가 분류 기준과 일치하는지 샘플링 검증

5. **상태 관리 경계 점검 (AR-02-02, AR-02-03)**
   - `domain/themes/ThemeProvider.tsx`, `ThemeContext.ts`, `useTheme.ts` — React Context가 domain에 있는 것이 적절한지 판단
   - `app/context/`, `app/hooks/` — UI 상태와 비즈니스 상태의 분리가 AR-02-03에 부합하는지

### 변경 범위

## 점검 결과 (2026-04-01)

### 1. 역방향 import (grep, `src/` 기준)

**`domain/` → `@app/` (허용 방향 위반 후보)**

| 파일 | import |
|------|--------|
| `domain/themes/ThemeProvider.tsx` | `@app/state/types`, `@app/state/palette-selection`, `@app/config/site-style`, `@app/components/GlobalSettings/types`, `@app/hooks/usePaletteResolution`, `@app/hooks/useCustomThemePresets` |
| `domain/themes/ThemeContext.ts` | `@app/state/types` |
| `domain/palettes/types.ts` | `@app/state/types` (re-export) |
| `domain/themes/e17-style-expressiveness.test.ts` | `@app/content/labs/token-lab/categories.json` |

**판단:** `ThemeProvider`가 앱 상태·팔레트 선택·GlobalSettings 타입·훅에 의존하는 구조라, 엄격한 `app → domain → shared` 단방향과 **불일치**한다. `docs/design/02-folder-structure.md`에는 `ThemeProvider.tsx`가 domain에 두어진 **예외**로 언급되어 있으나, **domain→app import** 자체는 AR-01-02 엄격 해석 시 위반으로 남는다. 장기적으로는 브리지 레이어(app에서 domain Theme 래핑) 또는 타입/상태 이동으로 역방향 제거를 검토할 수 있다.

**`shared/` → `@domain/` (허용 방향 위반 후보)**

| 파일 | import |
|------|--------|
| `shared/@types/theme.d.ts` | `@domain/constants/theme-presets` (`PaletteName` re-export) |

**판단:** ambient 타입에서 도메인 상수 타입을 끌어오는 1건. shared가 domain에 의존하므로 **역방향**에 해당한다. 완화책: `PaletteName`을 `shared` 또는 `domain` 경계 전용 타입 모듈로 옮기거나, ambient에서 중복 선언 최소화.

**`app/` → `@domain/`:** 다수(정상 허용 방향). 샘플은 `app/pages/labs/*`, `app/hooks/*`, `GlobalSettings` 등.

### 2. Barrel 우회 (샘플)

- `@app/components/.../...` 직접 경로 예: `App.tsx` → `ComponentInspector/ComponentInspectorPanel`, `ThemeProvider` → `GlobalSettings/types`.
- 전수 목록화는 이번 점검에서 생략; 필요 시 `rg "from '@app/components/" src`로 확장 가능.

### 3. 아키텍처 문서 (AR-01-01)

- **`docs/design/ARCHITECTURE.md`:** 프로젝트 목적, 토큰/팔레트 흐름, 기술 스택 등 **3-layer 선정의 맥락**이 서술되어 있음.
- **`docs/design/02-folder-structure.md`:** domain/app/shared **폴더 역할**과 `ThemeProvider` 예외 위치를 명시.

### 4. 컴포넌트 분류 (AR-02-01)

- **Atoms/Molecules/Organisms** 명칭은 문서에 없음.
- **`02-folder-structure.md`**가 `app/components/`, `pages/`, `layouts/` 등 **폴더 단위 분류**로 실제 구조와 대응함 → Atomic Design 대신 **폴더·계층 문서**가 사실상 분류 기준.

### 5. Context / hook 경계 (AR-02-03)

- **Theme:** `domain/themes/`에 `ThemeContext`, `ThemeProvider`, `useTheme` 존재. 문서상 예외로 허용.
- **앱 측:** `app/context/`, `app/hooks/`에서 팔레트·테마·UI 브리지 역할 — UI/도메인 경계는 **ThemeProvider의 domain→app 의존** 때문에 한쪽으로 기울어 있음(위 1번과 동일 이슈).

### 권장 후속 (코드 변경은 별도 에픽/백로그)

1. `domain` → `@app` 의존 제거 방안 설계(타입만 shared/공통 계약으로 이동 등).
2. `shared/@types/theme.d.ts`의 `@domain` 의존 제거 또는 계약 모듈 분리.
3. (선택) barrel 일관성 가이드 추가 후 점진적 정리.
